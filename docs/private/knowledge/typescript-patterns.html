TypeScript 패턴

CodeViz 코드에서 반복적으로 등장하는 TypeScript 패턴을 정리합니다.
TS에 익숙하다면 이 페이지를 건너뛰어도 됩니다.

1. 타입 선언: type과 interface

TypeScript의 핵심은 "이 변수에 어떤 형태의 데이터가 들어올 수 있는가"를 선언하는 것입니다.

type — 허용 값 나열 (Union Type)

type NodeKind = 'directory' | 'file' | 'function';
// NodeKind 변수는 이 세 문자열 중 하나만 가능
// "|"는 "또는". 수학의 합집합(∪)과 같은 개념

이것을 리터럴 유니온(literal union) 타입이라고 합니다. 문자열 리터럴뿐 아니라 숫자, 다른 타입도 조합할 수 있습니다:

type Result = string | null;          // 문자열이거나 null
type ID = string | number;              // 문자열이거나 숫자

interface — 객체의 형태 정의

interface GraphNode {
  id: string;              // 필수 필드
  lineStart?: number;      // ?는 "있을 수도 없을 수도" (optional)
  children?: GraphNode[];   // 자기 자신을 재귀 참조할 수 있음
}

// 사용 시:
const node: GraphNode = { id: 'file-1' };                // OK (optional 필드 생략)
const node2: GraphNode = { id: 'dir-1', lineStart: 10 };  // OK
const bad: GraphNode = { name: 'x' };                     // ERROR: id가 없음
type vs interface 차이: 실무적으로 거의 같습니다. 이 프로젝트에서는 객체 형태는 interface, 유니온/별칭은 type으로 구분합니다.

2. 제네릭 <T> — "나중에 정할 타입"

함수나 클래스가 어떤 타입이든 받을 수 있게 하면서도, 그 타입의 일관성은 유지하는 문법입니다.

// Map<K, V> — K 타입의 키, V 타입의 값을 갖는 딕셔너리
let astMap = $state<Map<string, ASTSymbol[]>>(new Map());
//                  ^^^^^^^^^^^^^^^^^^^^^^^^
// Map의 key는 string, value는 ASTSymbol 배열
// $state<T>(초기값) → Svelte 5의 반응형 상태를 T 타입으로 생성

// Array<T> 와 T[] 는 같은 의미:
let items: Array<string> = [];  // === string[]

제네릭은 "빈칸"이라고 생각하면 됩니다. Map<string, number>는 "key는 string, value는 number로 빈칸을 채운 Map"입니다.

3. 타입 단언과 Non-null 단언

as — "이 타입이라고 내가 보장한다"

const entry = child as FileSystemFileHandle;
// TS 컴파일러에게 "이건 FileSystemFileHandle 타입이야"라고 알려줌
// 실제 런타임 변환이 아니라 컴파일러 힌트. JS로 변환되면 사라짐

! — "null/undefined 아님을 보장"

const node = stack.pop()!;
// pop()은 undefined를 반환할 수 있지만 (빈 배열이면)
// !를 붙여 "이 시점에는 반드시 값이 있다"고 선언
// 실제로 없으면 런타임 에러가 발생하므로, 로직상 확실할 때만 사용

as any — 타입 시스템 우회

(node as any)._x = depth * INDENT_PX;
// _x는 GraphNode 타입에 정의되지 않은 임시 필드
// as any로 타입 검사를 우회. "나는 내가 뭘 하는지 안다"는 의미
// 가능하면 피하는 것이 좋지만, 레이아웃 임시 좌표 같은 경우에 사용

4. Record<K, V> — 딕셔너리 타입

const EXTENSION_MAP: Record<string, string> = {
  '.py': 'python',
  '.js': 'javascript',
  '.ts': 'typescript',
};
// Record<K, V> = 모든 key가 K 타입이고 value가 V 타입인 객체
// Python의 Dict[str, str]에 대응

5. import type — 타입만 가져오기

import type { GraphNode } from '$lib/types/graph';
// 런타임에는 완전히 사라짐 (JS 번들에 포함 안 됨)
// 타입 검사에만 사용되는 import
// → 번들 크기 최적화 + "이건 타입 용도로만 쓴다"는 의도 표현

import { buildGraph } from '$lib/services/graphBuilder';
// 일반 import: 실제 함수/값을 가져옴 (번들에 포함)

6. Promise와 async/await

이 프로젝트에서 비동기 작업(파일 읽기, API 호출, Worker 통신)이 많아 Promise 패턴이 자주 등장합니다.

Promise 기초

// Promise = "미래에 완료될 작업"을 나타내는 객체
function readFile(): Promise<string> {
  return new Promise((resolve, reject) => {
    // 성공 시: resolve(결과값)
    // 실패 시: reject(에러)
  });
}

async/await — Promise를 동기처럼 쓰기

async function load() {
  const result = await readFile();  // 완료될 때까지 대기
  console.log(result);              // resolve된 값을 받음
}
// await는 async 함수 안에서만 사용 가능
// try/catch로 reject된 에러를 잡음

이 프로젝트의 Promise 래핑 패턴

// parserService.ts — Worker에 메시지를 보내고 응답을 기다리는 패턴
function parseFile(path: string): Promise<ASTSymbol[]> {
  return new Promise((resolve) => {
    const id = nextId++;
    pendingRequests.set(id, resolve);  // resolve 함수를 Map에 저장
    worker.postMessage({ id, path });  // Worker에 요청 보냄
  });
  // Worker가 응답하면 onmessage에서 pendingRequests.get(id)로
  // 저장해둔 resolve를 꺼내어 호출 → Promise가 완료됨
}

7. Discriminated Union — 타입 좁히기

이 프로젝트의 Worker 통신, 인증 트랙 등에서 핵심적으로 사용되는 패턴입니다.

// 공통 필드(type)로 메시지 종류를 구분
interface InitMessage { type: 'init'; wasmUrl: string; }
interface ParseMessage { type: 'parse'; code: string; lang: string; }
type WorkerMessage = InitMessage | ParseMessage;

function handle(msg: WorkerMessage) {
  if (msg.type === 'init') {
    // 여기서 TS는 msg가 InitMessage임을 자동으로 알아냄 (narrowing)
    console.log(msg.wasmUrl);  // OK — InitMessage에만 있는 필드
  } else {
    // 여기서 msg는 ParseMessage
    console.log(msg.code);     // OK — ParseMessage에만 있는 필드
  }
}
왜 유용한가: Worker, WebSocket, API 응답 등 "여러 종류의 메시지가 하나의 채널로 오는" 상황에서 type 필드 하나로 안전하게 분기할 수 있습니다. if문 안에서 자동으로 타입이 좁혀지므로 잘못된 필드 접근을 컴파일 타임에 잡아줍니다.

8. 에러 처리의 unknown 타입

try {
  await callAPI();
} catch (error) {
  // TS에서 catch의 error는 기본적으로 unknown 타입
  // .message에 접근하려면 Error로 단언이 필요:
  return { content: `Error: ${(error as Error).message}` };
}
// 왜 unknown인가?
// throw는 어떤 값이든 던질 수 있기 때문 (throw "문자열"도 가능)
// TS는 안전하게 unknown으로 처리하고, 개발자가 타입을 확인하도록 요구

요약: 이 프로젝트에서 가장 많이 보이는 패턴

패턴등장 빈도대표 사용처
