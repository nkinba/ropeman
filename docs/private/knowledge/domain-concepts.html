도메인 지식

CodeViz가 해결하려는 문제, 핵심 도메인 개념, 그리고 데이터가 변환되는 과정을 설명합니다.

1. 문제 정의: 코드를 "읽지 않고" 이해하기

새로운 프로젝트를 파악하려면 보통 이런 과정을 거칩니다:

  1. README를 읽는다 (있으면)
  2. 디렉토리 구조를 훑어본다
  3. 진입점(main, index)을 찾아 읽기 시작한다
  4. import를 따라가며 파일을 하나씩 열어본다
  5. 머릿속으로 구조를 그려본다

CodeViz는 이 과정을 자동화합니다:

기존 방식: 눈으로 파일 읽기 → 머릿속에 구조 그리기 → 시간이 오래 걸림 CodeViz 방식: 폴더 선택 → 자동 파싱 → 그래프 시각화 → AI 아키텍처 분석 → 즉시 파악

2. 두 가지 관점: 물리적 구조 vs 논리적 구조

같은 프로젝트를 두 가지 관점으로 볼 수 있습니다. CodeViz는 두 가지를 모두 제공합니다.

Files 뷰 — 물리적 구조

"파일 시스템에 어떻게 배치되어 있는가?"를 보여줍니다.

nanochat/ ├── nanochat/ │ ├── model.py ← Transformer, Attention 클래스 │ ├── data.py ← ChatDataset, Tokenizer 클래스 │ ├── train.py ← train(), evaluate() 함수 │ └── generate.py ← sample(), generate() 함수 ├── scripts/ │ ├── train_gpt.py ← 학습 실행 스크립트 │ └── profile_train.py └── setup.py

이것은 IDE의 파일 탐색기와 동일합니다. 파일이 "어디에 있는지"는 알 수 있지만, "무슨 역할을 하는지"는 파일을 열어봐야 압니다.

Semantic 뷰 — 논리적 구조

"이 프로젝트의 각 부분이 어떤 역할을 하는가?"를 보여줍니다.

┌──────────────┐ ┌─────────────────┐ ┌──────────────┐ │ Data │────▶│ Model & │────▶│ Inference & │ │ Pipeline │ │ Training │ │ Generation │ │ │ │ │ │ │ │ data.py │ │ model.py │ │ generate.py │ │ ChatDataset │ │ train.py │ │ sample() │ │ Tokenizer │ │ Transformer │ │ ChatSession │ └──────────────┘ └─────────────────┘ └──────────────┘ │ ┌──────┴──────┐ │ Scripts & │ │ CLI │ │ │ │ train_gpt.py │ │ profile_... │ └─────────────┘

물리적으로 같은 디렉토리에 있는 파일들도 역할이 다르면 다른 블록에 배치됩니다. 반대로 다른 디렉토리의 파일들도 같은 역할이면 같은 블록에 묶입니다.

핵심 차이: Files 뷰는 readDirectoryRecursive()의 결과를 그대로 그래프로 변환합니다 (기계적). Semantic 뷰는 AI가 코드의 의미를 분석하여 역할 기반으로 재구성합니다 (지능적).

3. AST(Abstract Syntax Tree)와 심볼 추출

소스 코드는 텍스트, AST는 구조

프로그래밍 언어의 소스 코드는 사람에게는 읽기 쉽지만, 프로그램이 분석하려면 구조화된 형태로 변환해야 합니다. 이것이 파싱(parsing)이고, 결과물이 구문 트리(syntax tree)입니다.

# 이 Python 코드를:
class Transformer:
    def forward(self, x):
        return self.layers(x)

    def generate(self, prompt):
        pass
파서가 이렇게 변환: Module └── ClassDefinition ├── name: "Transformer" └── body: ├── FunctionDefinition │ ├── name: "forward" │ ├── parameters: ["self", "x"] │ └── body: ReturnStatement(...) └── FunctionDefinition ├── name: "generate" ├── parameters: ["self", "prompt"] └── body: PassStatement

CodeViz의 심볼 추출

CodeViz는 전체 AST가 아니라 최상위 심볼만 추출합니다. 즉, 클래스, 함수, 주요 변수 등 "이 파일에 뭐가 있는가"를 요약합니다:

// 위 코드에서 추출되는 ASTSymbol[]:
[
  { name: "Transformer", kind: "class", lineStart: 1, lineEnd: 8 },
  { name: "forward", kind: "function", lineStart: 2, lineEnd: 4, parent: "Transformer" },
  { name: "generate", kind: "function", lineStart: 5, lineEnd: 8, parent: "Transformer" },
]

심볼의 용도

용도설명
그래프 노드 생성각 심볼이 Files 뷰에서 노드(사각형)로 렌더링됨
스켈레톤 생성AI에게 보낼 프로젝트 요약에 심볼 목록 포함
시맨틱 노드 태그Semantic 뷰의 노드에 keySymbols로 표시 (예: "Transformer, train")
코드 미리보기NodeDetailPanel에서 심볼 위치 기반으로 코드 하이라이트

4. 스켈레톤(Skeleton) — AI에게 보내는 프로젝트 요약

AI에게 전체 소스 코드를 보내면 토큰 한도를 초과합니다. 대신 프로젝트의 뼈대만 요약하여 보냅니다. 이것이 Skeleton입니다.

// SkeletonPayload 예시 (skeletonExtractor.ts가 생성):
{
  "projectName": "nanochat",
  "totalFiles": 12,
  "files": [
    {
      "path": "nanochat/model.py",
      "language": "python",
      "symbols": [
        { "name": "Transformer", "kind": "class" },
        { "name": "Attention", "kind": "class" }
      ]
    },
    {
      "path": "nanochat/train.py",
      "language": "python",
      "symbols": [
        { "name": "train", "kind": "function" },
        { "name": "evaluate", "kind": "function" }
      ]
    }
  ]
}

AI는 이 요약만 보고 "이 프로젝트는 학습/데이터/추론/설정 역할로 나눌 수 있다"고 판단합니다.

서브 스켈레톤 (드릴다운용)

특정 시맨틱 노드를 드릴다운할 때는 해당 노드에 속한 파일들만으로 부분 스켈레톤을 생성합니다. extractSubSkeleton(filePaths, fileTree, astMap) 함수가 이 역할을 합니다.

"Model & Training" 노드 드릴다운 시: 전체 스켈레톤 (12 파일) │ filePaths 필터 ▼ 부분 스켈레톤 (model.py, train.py만) │ AI 분석 ▼ 하위 역할: "Model Architecture", "Training Loop", "Optimizer"

5. 시맨틱 분석 — AI가 하는 일

입력과 출력

// 입력: 스켈레톤 (위에서 설명한 프로젝트 요약)
// 시스템 프롬프트: "이 프로젝트를 3-7개의 의미론적 역할로 분류하라"

// AI 출력 (JSON):
{
  "roles": [
    {
      "label": "Model & Training",
      "description": "Core transformer model, training loops, and optimization",
      "filePaths": ["nanochat/model.py", "nanochat/train.py"],
      "keySymbols": ["Transformer", "train", "AdamW"]
    },
    // ... 3-7개 역할
  ],
  "edges": [
    { "source": "Data Pipeline", "target": "Model & Training", "type": "depends_on" }
  ]
}

관계(Edge) 유형

type의미예시
depends_onA가 B에 의존Training이 Data Pipeline에 의존
callsA가 B를 호출Scripts가 Training을 호출
extendsA가 B를 확장Custom Model이 Base Model을 확장
usesA가 B를 사용Inference가 Configuration을 사용

파싱과 정규화

AI 응답은 마크다운 코드블록으로 감싸져 있을 수 있고, 필드가 누락되거나 엣지 타입이 유효하지 않을 수 있습니다. parseSemanticLevel()이 이를 처리합니다:

  1. 마크다운 코드블록 (```json ... ```) 제거
  2. JSON 파싱
  3. 역할 ID에 sem: 프리픽스 부여 (다른 노드 ID와 충돌 방지)
  4. 엣지 type 유효성 검증 (4가지 외의 값 → uses로 폴백)
  5. SEMANTIC_COLORS 8색 순환 배정 (각 역할에 고유한 색상)

6. 재귀적 드릴다운

시맨틱 노드를 더블클릭하면 해당 노드의 내부 구조를 AI가 다시 분석합니다.

레벨 0 (최상위): ┌──────────┐ ┌──────────────┐ ┌──────────┐ │ Data │─▶│ Model & │─▶│ Inference│ │ Pipeline │ │ Training │ │ │ └──────────┘ └──────┬───────┘ └──────────┘ │ 더블클릭 (드릴다운) │ ▼ 레벨 1 ("Model & Training" 내부): ┌──────────────┐ ┌───────────────┐ ┌────────────┐ │ Model │─▶│ Training │─▶│ Optimizer │ │ Architecture │ │ Loop │ │ & Scheduler│ └──────────────┘ └───────────────┘ └────────────┘

드릴다운 흐름

  1. 사용자가 시맨틱 노드를 더블클릭
  2. semanticStore.drillDown(node) 호출
  3. 캐시 확인 (이전에 분석한 적 있는가?)
    • 캐시 히트: 즉시 하위 레벨 표시 (return true)
    • 캐시 미스: return false → AI 분석 트리거
  4. AI 분석: analyzeDrilldown(parentNode)
    • 부모 노드의 filePaths로 서브 스켈레톤 생성
    • AI에게 "이 파일들의 내부 하위 역할을 분류하라" 요청
    • 결과를 캐시에 저장
  5. 하위 레벨이 캔버스에 렌더링됨

브레드크럼 네비게이션

PROJECT ▸ Model & Training ▸ Training Loop ^^^^^^^ ^^^^^^^^^^^^^^^^ ^^^^^^^^^^^^^ 레벨 0 레벨 1 현재 위치 ← 클릭하면 해당 레벨로 되돌아감

semanticStore.drilldownPath 배열이 경로를 추적하고, Breadcrumb.svelte가 이를 클릭 가능한 경로로 렌더링합니다.

7. 양방향 연동 (Bidirectional Linking)

Semantic 뷰의 다이어그램과 File Explorer 사이드바는 서로 연결되어 있습니다. 이것은 "추상적 역할"과 "구체적 파일"을 즉시 매핑할 수 있게 해줍니다.

방향 1: 시맨틱 노드 → Explorer 파일 하이라이트

사용자가 "Data Pipeline" 노드 클릭 │ ▼ semanticStore.selectedSemanticNode = "Data Pipeline" │ ▼ (derived) semanticStore.highlightedFilePaths = ["nanochat/data.py", "nanochat/tokenizer.py"] │ ▼ (FileExplorer가 구독) FileExplorer에서 data.py, tokenizer.py에 노란 배경 + 자동 스크롤

방향 2: Explorer 파일 → 시맨틱 노드 하이라이트

사용자가 Explorer에서 "train.py" 클릭 │ ▼ semanticStore.findSemanticNodeForFile("nanochat/train.py") │ → currentLevel.nodes에서 filePaths에 "train.py"를 포함하는 노드 탐색 ▼ semanticStore.selectedSemanticNode = "Model & Training" │ ▼ (ZUICanvas Effect 1a가 감지) "Model & Training" 노드에 data.highlighted = true │ ▼ (SemanticNode.svelte) 파란색 글로우 효과 표시

핵심 데이터: filePaths

양방향 연동의 핵심은 SemanticNode.filePaths 필드입니다. 각 시맨틱 노드가 어떤 파일들에 대응하는지를 이 배열이 기록하고 있어서, 노드→파일 방향(정방향)과 파일→노드 방향(역방향) 조회가 모두 가능합니다.

9. 캐싱 전략

이 프로젝트는 두 가지 수준의 캐싱을 사용합니다:

시맨틱 레벨 캐시 (인메모리)

위치semanticStore.cache (Map)
키'__root__' (최상위) 또는 노드 ID (드릴다운 레벨)
값SemanticLevel 객체 (nodes + edges + breadcrumbLabel)
수명세션 동안 유지 (페이지 새로고침 시 초기화)
목적드릴다운 왕복 시 AI 재호출 방지

AI 응답 캐시 (영속)

위치Orama + IndexedDB
키질문 텍스트의 벡터 임베딩
값AI 응답 텍스트
수명브라우저 저장소에 영속 (수동 삭제 전까지)
목적유사한 질문에 대해 API 호출 생략

10. 인증 트랙: BYOK vs Bridge

AI 기능을 사용하려면 LLM에 접근해야 합니다. 두 가지 경로가 있습니다:

BYOK (Bring Your Own Key)Bridge (로컬 브릿지)
방식사용자가 Gemini API 키 직접 입력로컬에서 실행 중인 브릿지 서버에 WebSocket 연결
API 호출브라우저 → Gemini API (HTTPS)브라우저 → localhost WebSocket → 로컬 LLM
비용Gemini API 사용료무료 (로컬 실행)
설정API 키만 입력브릿지 서버 별도 설치 필요
authStore.activeTrack이 현재 활성 트랙을 결정합니다. 우선순위: Bridge 연결됨 > API 키 있음 > none. AI 서비스들(aiService, semanticAnalysisService)은 이 트랙에 따라 API 호출 경로를 분기합니다.
