프론트엔드 기술 스택

CodeViz가 의존하는 핵심 기술들의 역할, 관계, 그리고 코드에서 어떻게 나타나는지 설명합니다.

1. SvelteKit — 앱 프레임워크

Svelte와 SvelteKit의 관계

Svelte = UI 컴파일러. .svelte 파일 → 순수 JS로 변환 SvelteKit = Svelte + 라우팅 + 빌드 시스템 + SSR 프레임워크 대응 관계: React → Next.js Vue → Nuxt.js Svelte → SvelteKit ← 이 프로젝트

Svelte의 핵심 차이점: React/Vue는 런타임에 가상 DOM을 비교(diffing)하지만, Svelte는 빌드 시점에 컴파일하여 DOM을 직접 조작하는 코드를 생성합니다. 따라서 런타임 라이브러리가 필요 없고, 번들이 작고 빠릅니다.

SvelteKit의 파일 기반 라우팅

src/
├── routes/
│   ├── +layout.svelte      // 모든 페이지에 적용되는 글로벌 레이아웃
│   └── +page.svelte        // "/" 경로의 페이지 컴포넌트
├── lib/                    // $lib 별칭으로 접근하는 라이브러리 코드
│   ├── components/         // UI 컴포넌트
│   ├── stores/             // 상태 관리
│   ├── services/           // 비즈니스 로직
│   └── types/              // TypeScript 타입
└── app.html                // HTML 템플릿 (SvelteKit이 관리)
이 프로젝트의 특수성: adapter-static을 사용하여 SSR(서버 사이드 렌더링) 없이 순수 정적 사이트로 빌드합니다. 서버 코드가 전혀 없으며, 모든 로직이 브라우저에서 실행됩니다. 페이지도 +page.svelte 하나뿐인 SPA(Single Page Application)입니다.

$lib 경로 별칭

// 코드에서 이런 import를 자주 보게 됩니다:
import { graphStore } from '$lib/stores/graphStore.svelte';

// $lib = src/lib/의 별칭 (SvelteKit이 자동 설정)
// '../../../lib/stores/...' 같은 상대경로를 피할 수 있음

2. Svelte 5 Runes — 반응형 시스템

Svelte 5에서 도입된 새로운 반응형 프리미티브입니다. 이 프로젝트의 모든 상태 관리가 runes 기반이므로 이해가 필수적입니다.

핵심 4개 rune

Rune역할React 대응실행 시점
$state(값)반응형 상태 선언useState초기화 시 1회
$derived(표현식)파생 값 (자동 재계산)useMemo의존값 변경 시
$effect(() => ...)부수효과 실행useEffect의존값 변경 시
$props()컴포넌트 속성 받기함수 인자부모 렌더 시

$state — 반응형 상태

let count = $state(0);
// count가 변경되면 이를 읽는 모든 곳이 자동으로 업데이트

count = 5;  // 단순 할당으로 상태 변경 (React의 setCount(5)에 대응)

$derived — 파생 값

let count = $state(0);
const doubled = $derived(count * 2);
// count가 3이면 doubled는 자동으로 6
// 명시적 의존성 배열이 필요 없음 — Svelte가 자동 추적

$effect — 부수효과

$effect(() => {
  console.log(`count is now ${count}`);
  // count를 읽으므로 count 변경 시 자동 재실행
  // React useEffect와 달리 의존성 배열을 직접 쓸 필요 없음
});

// 이 프로젝트의 ZUICanvas.svelte에서:
$effect(() => {
  const level = semanticStore.currentLevel;  // 의존성 자동 추적
  if (!level) return;
  nodes = toSemanticFlowNodes(level);         // level 변경 시 노드 재생성
});

$state.raw — 성능 최적화

let nodes = $state.raw<Node[]>([]);
// 일반 $state는 깊은(deep) 반응성 — 배열 내부 요소 변경도 감지
// $state.raw는 참조(reference)만 추적 — 재할당만 감지
// 노드/엣지 배열처럼 큰 데이터에 사용하여 불필요한 Proxy 오버헤드 방지

nodes = [...nodes, newNode];  // 재할당 → 반응 O
nodes[0].x = 100;             // 내부 변경 → 반응 X (raw이므로)
Svelte 5 Proxy 주의점: $state로 만든 값은 JavaScript Proxy 객체로 감싸집니다. .sort(), .reverse(), .splice() 같은 in-place 변경 메서드를 호출하면 state_unsafe_mutation 에러가 발생합니다. 
// 에러 발생:
{#each node.children.sort(compareFn) as child}

// 해결: 스프레드로 복사 후 정렬
{#each [...node.children].sort(compareFn) as child}

.svelte.ts 파일 확장자

// Runes($state, $derived 등)는 두 곳에서만 사용 가능:
// 1. .svelte 컴포넌트 파일
// 2. .svelte.ts 모듈 파일 ← 스토어에서 사용

// 일반 .ts 파일에서는 $state를 쓸 수 없음
// 그래서 스토어 파일이 xxxStore.svelte.ts 확장자를 가짐

이 프로젝트의 스토어 패턴

Svelte 5에서는 runes를 활용한 클로저 싱글턴 패턴으로 상태를 관리합니다. 이 패턴이 모든 스토어에서 동일하게 반복됩니다:

// src/lib/stores/xxxStore.svelte.ts
function createXxxStore() {
  let value = $state(초기값);        // 반응형 상태 (외부에서 직접 접근 불가)

  return {
    get value() { return value; },   // getter로 읽기 노출
    set value(v) { value = v; },     // setter로 쓰기 노출
    someMethod() { ... },            // 복잡한 상태 변경 로직
  };
}

export const xxxStore = createXxxStore(); // 모듈 레벨 싱글턴
// → 어디서 import해도 같은 인스턴스
왜 클로저인가: $state 변수를 함수 스코프 안에 가두고, getter/setter만 노출합니다. 외부에서 xxxStore.value = 5로 변경하면 setter가 호출되어 반응형 업데이트가 트리거됩니다. 이것은 React의 Context + useReducer나 Zustand의 create()와 유사한 역할입니다.

3. Vite — 빌드 도구

Vite = 개발 서버 (ESM 네이티브, HMR) + 프로덕션 번들러 (Rollup 기반) 대응 관계: Create React App / webpack → Vite ← 이 프로젝트 Next.js (내장 번들러) → SvelteKit + Vite

개발 시 (Development)

  • 브라우저의 네이티브 ES Modules를 활용하여 번들링 없이 파일을 직접 서빙
  • 파일 변경 시 HMR(Hot Module Replacement)로 밀리초 단위 반영. 전체 빌드 불필요
  • npm run dev → http://localhost:5173 (또는 다음 빈 포트)

빌드 시 (Production)

  • npm run build → Rollup 기반 번들링: 트리쉐이킹, 코드 스플리팅, minification
  • 결과물은 build/ 폴더에 정적 파일로 출력 (adapter-static)

이 프로젝트에서 Vite가 하는 특수한 일

// vite.config.js에서 주목할 설정들:

// 1. $lib 경로 별칭 — SvelteKit이 자동 설정
//    import { x } from '$lib/stores/...'

// 2. WASM 파일 서빙
//    static/ 폴더의 .wasm 파일을 개발 서버가 그대로 서빙
//    tree-sitter-python.wasm, tree-sitter-javascript.wasm 등

// 3. ?testDir= dev 미들웨어
//    개발 시 ?testDir=/tmp/micrograd 파라미터로 로컬 디렉토리를 읽어
//    File System Access API 없이 테스트 데이터를 제공
import.meta.env: Vite가 제공하는 환경 변수 접근법입니다. import.meta.env.DEV는 개발 모드에서 true, 프로덕션에서 false. 이 프로젝트에서 dev-only 디버그 코드를 분기할 때 사용합니다.

4. @xyflow/svelte (SvelteFlow) — 그래프 렌더링

SvelteFlow = 노드 + 엣지 기반 인터랙티브 다이어그램 라이브러리 React Flow의 Svelte 포팅 용도: 플로차트, 마인드맵, 데이터 파이프라인 시각화, 코드 구조 시각화 등

핵심 개념

개념설명이 프로젝트에서
Node캔버스 위의 사각형 요소파일, 함수, 시맨틱 역할 블록
Edge노드 간 연결선import 관계, 의존성, 호출 관계
Handle노드의 연결 포인트top/bottom (TB 레이아웃), left/right (LR 레이아웃)
Viewport현재 보이는 캔버스 영역pan(드래그), zoom(스크롤) 상태
nodeTypes커스텀 노드 타입 등록 맵5종의 커스텀 노드 컴포넌트 매핑

데이터 형태

// SvelteFlow가 기대하는 노드 데이터:
{
  id: 'file-1',
  type: 'fileNode',              // nodeTypes 맵의 키 → 렌더링할 컴포넌트 결정

  data: { label: 'main.py', ... } // 커스텀 데이터 (컴포넌트가 props로 받음)
}

// SvelteFlow가 기대하는 엣지 데이터:
{

  source: 'file-1',               // 출발 노드 id
  target: 'file-2',               // 도착 노드 id
  type: 'smoothstep',             // 엣지 곡선 스타일

5. web-tree-sitter — 브라우저에서 코드 파싱

tree-sitter란?

tree-sitter = 증분(incremental) 파서 생성기 문법 파일 (.grammar.js) │ ┌───────┴───────┐ ▼ ▼ 네이티브 파서 WASM 파서 (C 라이브러리) (브라우저용) │ │ ▼ ▼ CLI 도구 web-tree-sitter ← 이 프로젝트 (에디터 등) (브라우저에서 실행)
  • 원래 GitHub의 코드 분석을 위해 만들어진 C 라이브러리
  • VSCode, Neovim 등 에디터의 구문 강조, 코드 접기 등에 사용
  • web-tree-sitter는 이를 WebAssembly(WASM)으로 컴파일한 버전

파싱 흐름

// 1. WASM 초기화 (한 번만)
await Parser.init();
const parser = new Parser();

// 2. 언어별 문법 로드
const Python = await Parser.Language.load('tree-sitter-python.wasm');
parser.setLanguage(Python);

// 3. 소스 코드 파싱 → CST(Concrete Syntax Tree)
const tree = parser.parse(`def add(a, b):\n    return a + b`);

// 4. 트리 순회하며 심볼 추출
tree.rootNode.children.forEach(node => {
  if (node.type === 'function_definition') {
    // → ASTSymbol { name: 'add', kind: 'function', ... }
  }
});

Web Worker에서 실행하는 이유

WASM 파싱은 CPU 집약적 작업입니다. 수백 개 파일을 메인 스레드에서 파싱하면 UI가 멈춥니다. 이 프로젝트에서는 Web Worker(별도 스레드)에서 파싱을 실행하고, postMessage/onmessage로 메인 스레드와 통신합니다. 
// 메인 스레드 (parserService.ts)
worker.postMessage({ type: 'parse', code, lang });

// Worker 스레드 (parserWorker.ts)
self.onmessage = (e) => {
  const symbols = parse(e.data.code, e.data.lang);
  self.postMessage({ symbols });
};

CST vs AST

CST (Concrete Syntax Tree)AST (Abstract Syntax Tree)
포함 내용모든 토큰 (괄호, 콤마, 키워드 등 포함)의미있는 구조만 (키워드, 구두점 제거)
tree-sitter 출력CST를 반환-
이 프로젝트CST에서 최상위 심볼만 추출하여 ASTSymbol[]로 변환이 변환 결과를 편의상 "AST"라고 부름

6. @dagrejs/dagre — 자동 그래프 레이아웃

dagre란?

dagre = Directed Acyclic Graph Rendering Engine = Graphviz의 "dot" 레이아웃 알고리즘을 JS로 구현 입력: 노드 목록 + 엣지 목록 + 옵션 출력: 각 노드의 (x, y) 좌표 ← 겹치지 않게 자동 배치

사용 방법

import dagre from '@dagrejs/dagre';

const g = new dagre.graphlib.Graph();
g.setGraph({
  rankdir: 'LR',   // 배치 방향: LR(좌→우), TB(위→아래), RL, BT
  nodesep: 80,      // 같은 계층 노드 간 간격 (px)
  ranksep: 120,     // 계층 간 간격 (px)
});

// 노드 등록 (크기 필수)
g.setNode('A', { width: 280, height: 160 });
g.setNode('B', { width: 280, height: 160 });

// 엣지 등록
g.setEdge('A', 'B');

// 레이아웃 계산
dagre.layout(g);

// 결과: 각 노드의 중심 좌표
g.node('A');  // { x: 140, y: 80, width: 280, height: 160 }

이 프로젝트에서의 사용처

파일방향용도
graphBuilder.tsTB (위→아래)아키텍처 그룹 레이아웃
semanticGraphBuilder.tsLR (좌→우)시맨틱 역할 다이어그램 레이아웃
dagre의 한계: dagre는 좌표를 계산만 하고, 렌더링은 하지 않습니다. 계산된 좌표를 SvelteFlow의 position 필드에 넣어주면 SvelteFlow가 렌더링합니다. 또한 dagre는 매 호출마다 처음부터 레이아웃을 재계산하므로, 사용자가 노드를 드래그해도 다음 effect 실행 시 위치가 리셋됩니다.

7. File System Access API — 브라우저에서 로컬 파일 접근

개요

// 사용자에게 폴더 선택 다이얼로그 표시

// 디렉토리 내용 순회 (비동기 이터레이터)
for await (const entry of dirHandle.values()) {
  if (entry.kind === 'file') {
    const file = await entry.getFile();
    const text = await file.text();  // 파일 내용 읽기
  } else if (entry.kind === 'directory') {
    // 재귀적으로 하위 디렉토리 순회
  }
}

8. Orama — 클라이언트 사이드 검색 엔진

개요

Orama = 브라우저/Node에서 실행되는 벡터 + 전문(full-text) 검색 엔진 전통적 검색: 서버 DB에 쿼리 → 결과 반환 Orama: 브라우저 IndexedDB에 벡터 저장 → 클라이언트에서 직접 검색

이 프로젝트에서의 역할

AI 응답을 캐싱하는 데 사용됩니다. 정확히 같은 질문이 아니라 유사한 질문에 대해 이전 응답을 재사용하기 위해 벡터 유사도 검색을 수행합니다:

  1. 사용자 질문 → Gemini embedding API로 벡터 변환
  2. Orama DB에서 코사인 유사도로 가장 가까운 벡터 검색
  3. 유사도가 임계값 이상이면 캐시된 응답 반환 (API 호출 생략)
  4. 임계값 미만이면 새로 API 호출 → 결과를 Orama에 저장
